# Pitch Outline — Universal Presentation Structure

This document outlines the 18 essential sections for presenting your solution to the judges. Customize each section to align with your specific challenge track.

## 1. Problem Statement
- **What**: Define the core challenge clearly.
- **Context**: Where does this occur? Why is it a problem now?

## 2. Target User
- **Who**: Who suffers from this issue? Define the direct user persona.

## 3. Current Process
- **How**: Describe how the problem is currently handled (manual, legacy software, etc.).

## 4. Business/Social Impact
- **Cost**: What are the consequences of not solving this problem? (Time lost, financial impact, safety risks).

## 5. Proposed Solution
- **What**: High-level pitch of your system solution. Keep it neutral and outcome-focused.

## 6. Rationale for Selected Technique
- **Why**: Explain why you chose this specific engineering approach (e.g. heuristics, linear solver, forecasting, or agentic flow). Avoid generic "AI makes it better" claims.

## 7. Baseline Reference
- **Baseline**: What is the simplest fallback (e.g., standard SQL rules, spreadsheet formulas, random guess, average statistic)? Show you understand simpler methods.

## 8. Live Demonstration
- **Show**: Run the capability on the Workspace UI live, showing the input file upload, running processing state, and output artifact rendering.

## 9. System Architecture
- **Structure**: Walk through your system components (frontend SPA, API gateways, core processors, local persistence folders).

## 10. Data Inventory
- **Inputs**: Details on datasets used, processing formats, size, and schemas.

## 11. Evaluation Methodology
- **Scoring**: How do you measure success? Show your metric registry (accuracy, latency, solving duration, error rates).

## 12. Results & Performance
- **Scores**: Show actual report figures generated by the evaluation runner (success rate, average runtime).

## 13. Security & Safety
- **Defense**: Highlight protection against path traversals, file type exclusions, and automatic credential scrubbing in logs.

## 14. Implementation Feasibility
- **Delivery**: Explain how this plugs into existing pipelines. Show that docker-compose is immediately ready to run offline or in the cloud.

## 15. Pilot Criteria
- **Validation**: How will you validate the system in a limited trial? Define go/no-go boundaries.

## 16. Future Scalability
- **Growth**: How does the system handle 10x-100x volume (e.g., introducing task queues, cache layers, distributed databases)?

## 17. Business & Social Value
- **ROI**: Estimate value returned to the target user/organization.

## 18. Conclusion
- **Wrap-up**: Summary statement linking technical outcome back to the core user problem.
