## 1. Academic Integrity & Algorithm Alignment

### 1.1 Elo (Pelánek 2016) — mostly sound, one citation overreach

**Correct:** Logistic expected score `1/(1+10^((θ−α)/400))`, symmetric student/question update, overflow clamp on the exponent (`±20`), and the dynamic K-factors (`K_student = max(16, 48/(1+n/10))`, `K_question = max(8, 32/(1+n/20))`) are all faithful to Pelánek's uncertainty-decreasing-K recommendation. Time-gap K bump and AI-help K=0 freeze are reasonable engineering extensions.

**Problems:**

- **Citation overreach on "Response-Time Weighted Elo."** The `academic-citations.mdx` and the plan attribute speed-weighting to Pelánek (2016). Pelánek (2016) does **not** propose response-time weighting of the score `S`. This is closer to van der Linden's response-time models or the BKT+time literature. Attributing it to Pelánek is inaccurate — either cite the correct source or label it explicitly as a novel heuristic.
- **The speed factor silently leaks into the student rating.** In the SQL, `v_speed_factor` multiplies `v_sd` (student delta) as well as `v_qd` (question delta), and `v_sd` is computed *before* the student-Elo update. So a fast correct answer inflates the student's ability gain — but only when the answer is correct, only in the non-AI branch. This asymmetry (fast-and-right is rewarded, fast-and-wrong is not penalized symmetrically) is undocumented and not in any cited paper. It biases ratings upward for quick guessers on easy items.
- **Hint discount and speed factor stack multiplicatively** on the same `v_sd` with no documented interaction, making the effective K hard to reason about.

### 1.2 BKT (Corbett & Anderson 1994) — correct core, non-standard partial-credit, dead branch

**Correct:** The binary posterior updates match Corbett & Anderson exactly:
`P(L|correct)=P·(1−S)/(P(1−S)+(1−P)G)`, `P(L|incorrect)=P·S/(P·S+(1−P)(1−G))`, followed by the learning transition `P' = post + (1−post)·T`. Clamping to `[0.0001, 0.9999]` is fine.

**Problems:**

- **The partial-credit branch (`0 < score < 1`) is invented and uncited.** It blends the correct/incorrect posteriors by `score·post_correct + (1−score)·post_incorrect`. Classic BKT is strictly binary; this heuristic has no grounding in Corbett & Anderson and changes the model's likelihood semantics. It is also **dead code in production** — `grade_answer()` only ever returns `0.0` or `1.0` for mcq/numeric/short_answer — so it is never exercised by the API but *is* exercised by nothing, meaning it's untested and misleading. Decide: either remove it, or cite a fuzzy/partial-credit BKT variant (e.g., Wang & Heffernan) and test it.
- **Parameter source asymmetry.** Production reads per-concept `bkt_parameters` from the DB (good), but falls back to `T=0.10`. The eval uses `T=0.06`. Pick one canonical default and document it.

### 1.3 LinUCB (Li et al. 2010) — Sherman–Morrison correct, but split-brain and fragile

**Verified numerically:** I reproduced the SQL update `A⁻¹ ← A⁻¹ − (w wᵀ)/(1 + xᵀw)`, `w = A⁻¹x`, against `numpy.linalg.inv(A + xxᵀ)` — **it matches exactly.** The rank-1 update is mathematically correct.

**Problems:**

- **Split-brain bandit.** The arm matrix `A⁻¹` and vector `b` are **updated in SQL** but the arm is **selected in Python** (`bandit.select_arm` in `elo.py`/`bandit.py`, not in the package). Two codebases must agree byte-for-byte on the LinUCB formulation forever, or the policy that picks arms diverges from the policy that learns. This is a maintenance landmine.
- **Numerical hygiene is missing.** No symmetrization of `A⁻¹` after the update, no periodic re-inversion from `A`, and no floor on conditioning. Over thousands of `numeric` rank-1 updates, `A⁻¹` will lose symmetry and positive-definiteness; the UCB term `√(xᵀA⁻¹x)` can then go imaginary and crash arm selection.
- **Silent state desync.** When `denom ≤ 1e-6` the code skips the `A⁻¹` update **but still updates `b`**. `A⁻¹` and `b` then describe different histories — `θ̂ = A⁻¹b` becomes meaningless for that arm.
- **`numeric` for matrix algebra.** PostgreSQL `numeric` is arbitrary-precision and ~10–100× slower than `float8`. Inside a write-locked critical section (see §3) this lengthens every lock hold. Use `double precision`.

### 1.4 "FSRS" Forgetting — **this is not FSRS** (quantified)

The implemented decay is `P_eff = P_stored · 2^(−Δt/S)` — a **pure exponential** (Ebbinghaus / Woźniak two-component simplification). Real FSRS (v4/v4.5/v5) uses a **power-function** retrievability `R(t) = (1 + FACTOR·t/S)^DECAY` (FACTOR=19/81, DECAY≈−0.5). They diverge sharply:

| Δt (days), S=4 | Implemented `2^(−t/S)` | Actual FSRS power curve |
|---|---|---|
| 2 | 0.707 | 0.946 |
| 4 | 0.500 | 0.900 |
| 8 | 0.250 | 0.825 |

The implemented curve forgets **3–4× faster** than FSRS. Calling it FSRS in `academic-citations.mdx` is a misattribution. Two honest options: (a) rename it "exponential forgetting (Ebbinghaus/Woźniak)" and cite accordingly — your Woźniak 1994 citation already supports this; or (b) implement the actual FSRS power function if you want the FSRS name.

Additional bug: **stability grows unbounded.** `stability_days *= 2` on every score ≥ 0.8 with no cap. I simulated 20 successful reviews → **S ≈ 3.15 million days**, at which point `2^(−t/S) ≈ 1` for any realistic `t` and **forgetting is effectively disabled**. Cap stability (FSRS caps at ~36500 days) and use a smoother stability-increment than naive doubling.

### 1.5 Graph propagation (Hwang 2003 / Graphusion) — path-dependent and divergent

- **Production (SQL) is depth-1; reference (Python) is unbounded recursion.** Different algorithms, same name. exp4 validates the recursive one; production runs the shallow one.
- **Propagation is path-dependent / not well-defined on DAGs with diamonds.** The `visited` set means a concept reachable by two prerequisite paths receives only the *first* path's delta (DFS-order dependent). Two different valid traversal orders produce different mastery states. For a system that claims mathematical rigor, the propagation operator should be order-invariant (e.g., aggregate all incoming deltas, then apply once).
- **Changelog mislabels test outputs as coefficients.** The changelog states "forward β = 0.425, backward γ = 0.3312." Those are the **expected B and C node values** from exp4's assertions, not decay coefficients. The actual coefficients are `BETA = GAMMA = 0.25` in both code paths. I confirmed `0.30 + 0.25·0.50 = 0.425` and `0.30 + 0.25·(0.425−0.30) = 0.3312`. Fix the changelog — as written it implies coefficients you do not use.

## 2. Evaluation Suite Verification

Strong points worth keeping: deterministic seeded RNG, Box–Muller with a `log(0)` guard, Common Random Numbers in exp3 (the per-`(student,trial)` seed is genuinely shared across LinUCB/Random/Greedy — good causal hygiene), CSV+PNG artifacts, and CI gate assertions.

But the suite is **not yet scientifically rigorous or publishable.** Issues by experiment:

### 2.1 exp2 BKT "AUC 0.994" is circular and measures the wrong thing
- **Well-specified to the point of tautology.** Responses are generated by `bkt_only=True` using the *exact same* guess/slip the estimator uses (`G=0.20, S=0.10`). You are recovering a model from data generated by that identical model with identical parameters. High AUC is guaranteed and tells you nothing about robustness.
- **AUC is computed against a latent label that only exists in simulation.** `y_true` is the unobservable true mastery state. On real data this number cannot be reproduced. A **publishable** BKT metric is **next-step predictive AUC on observed correctness** (predict response *t+1*, score against the actual response), reported with held-out students.
- **Pooling across all timesteps inflates separability.** Late-trajectory estimates are trivially separated; pooling them with early ones flatters the AUC.

### 2.2 exp3 "Regret Curves" — there is no regret here
- The code plots **cumulative reward**, not regret. **Regret requires an oracle** (the best arm in hindsight or the Bayes-optimal ZPD policy): `Regret(T) = Σ (r*_t − r_t)`. There is no oracle, so the headline "regret curve" claimed in the docs does not exist. Add an oracle that always selects the item with true `P(success)` closest to 0.75 and plot cumulative regret vs. it.
- **The reward is partly self-referential.** `calculate_bandit_reward(expected_success, score)` uses the model's *own* `expected_success`. A policy that drives `expected_success` toward target is rewarded by construction. Report the **realized** ZPD hit-rate (you already have `p_success_elo`) as the primary metric and keep reward as secondary.
- **No variance.** Single master seed, 30 students, 100 trials. No confidence intervals. Publishable BC results need **N≥30 seeds**, mean ± 95% CI (or bootstrap), and a significance test on the LinUCB-vs-baseline gap.
- **ZPD band inconsistency:** target is 0.75 everywhere, but the hit band is `[0.60, 0.85]` (asymmetric, midpoint 0.725). Justify or symmetrize.

### 2.3 exp1 Elo convergence — weak gate, confounded baseline
- **The RMSE<250 gate is barely better than trivial.** True abilities are `N(1500, 200)`. A model that simply guesses the population mean (1500) scores RMSE ≈ 200. Your threshold of 250 can pass a near-useless estimator. Gate on RMSE relative to the std (e.g., < 0.5·σ ≈ 100) or report normalized RMSE.
- **Random vs. Adaptive isn't a clean paired comparison.** Although true abilities are cloned, the shared global `rng` is consumed in different order across the two loops, so the response coin flips differ. Use CRN here as you did in exp3.
- **No identifiability/parameter-recovery test.** Add a test that recovers known true difficulties from simulated responses and reports correlation(est, true).

### 2.4 exp4 graph & exp5 forgetting — these are unit tests, not evaluations
- exp4 is a deterministic mock-DB assertion of hand-computed values (legitimate as a unit test) but uses `weight=1.0` everywhere and a 40-node *timing* check that measures an in-memory mock (`<50ms` is meaningless — no DB round-trips). Note also `db.get_concept_relations` is re-fetched on **every** recursion → O(N²) calls; harmless on a mock, catastrophic against a real DB.
- exp5 verifies the decay arithmetic (correct) and a "100 students/15 days" curve where all 100 start identical and only stability varies deterministically — there is **no empirical retention data** behind it. To be publishable, fit/validate the forgetting curve against real review logs (recall vs. elapsed time) and report calibration error.

### 2.5 What "publishable / scientifically provable" requires (concrete)
1. **Test the production code path, not the Python twin.** Run the eval harness through `submit_attempt_v3` (e.g., against an ephemeral Postgres in CI) or formally prove the Python and SQL implementations equivalent and assert it in a test.
2. **Multi-seed, CI-reported metrics:** ≥30 seeds, mean ± 95% CI, effect sizes, significance tests.
3. **Predictive metrics on held-out observable data:** next-step AUC/log-loss/Brier for BKT and Elo; reliability diagrams for calibration of `expected_success`.
4. **True regret vs. an oracle** for the bandit; sensitivity to `α`.
5. **Model misspecification tests:** generate with parameters ≠ estimator's, show graceful degradation.
6. **Real-data validation** on at least one public ITS dataset (e.g., ASSISTments) for external validity.

## 3. Production Feasibility — Red-Team

### 3.1 Database locks & latency under 1,000 concurrent submits — **this is the hard ceiling**

The transaction acquires, in order:
1. `SELECT * FROM app.student_concept_mastery ... FOR UPDATE` — per-student row, low contention.
2. `SELECT * FROM app.questions WHERE id = p_question_id FOR UPDATE` — **a single global row per question**.
3. `SELECT * FROM audit.bandit_arms ... FOR UPDATE` — **a single global row per (policy, question)**.

**Failure mode:** When 1,000 students answer the *same* popular question concurrently, steps 2 and 3 are **two serialization points on the same logical object**. Every transaction must take the question's row lock, do all the Elo+BKT+Sherman–Morrison `numeric` math, write audit rows, run propagation, and only then commit and release. Throughput on that question collapses to `1 / (transaction_hold_time)`. If the critical section is ~10–20 ms, you cap at **~50–100 submits/sec for that question regardless of how many CPU cores or replicas you add.** Concurrency does not help a serialized hot row.

**Pool starvation:** Each of the 999 waiting transactions **holds its database connection while blocked** on the lock. Supabase's pooler has a bounded connection count (often 15–60 on smaller tiers). You exhaust the pool almost immediately; new requests get connection timeouts, and the symptom looks like a total outage, not just slow question calibration. CPU will look *underused* while everything waits on locks — the classic lock-bound profile.

**Deadlock surface:** The propagation loops take `FOR UPDATE` on related-concept mastery rows with **no `ORDER BY`**, so lock-acquisition order is query-plan dependent. Concurrent submits by the same student (double-submit, multi-tab) on prerequisite-linked concepts can acquire locks in opposite orders → deadlock; Postgres kills one transaction. Cross-student conflicts are limited (mastery rows are per-student) but the question + bandit rows are shared.

**Verdict:** The atomic-everything-in-one-transaction design is correct for *consistency* and *replay protection*, but it makes question calibration and bandit learning a global serialization bottleneck. It will not sustain 1,000 concurrent submits on a hot item. Mitigation in §4.

### 3.2 Sherman–Morrison in PL/pgSQL — works, doesn't scale *here*

The 3×3 algebra itself is trivial CPU (a few dozen flops). The problem is **where** it runs: inside the question/arm critical section, in slow `numeric`, extending the lock hold. It also lacks the numerical safeguards (§1.3). It is not "too much math for the DB" in FLOP terms; it is **the wrong place** to do learning that contends on a hot row. Move bandit learning out of the synchronous transaction (§4).

### 3.3 Edge cases & failure modes

- **Extreme speed ratios / zero time:** `v_speed_ratio = response_time_ms / avg_time`. `response_time_ms = 0` (instant/automated) → ratio 0 → `speed_factor = min(1.2, 1+0.2) = 1.2` → **maximum reward boost for a 0-ms answer.** This rewards bots and double-clicks. `avg_response_time_ms` is guarded against 0, good, but the *client-supplied* time has only `ge=0` validation — no upper clamp, no sanity floor. Reject/winsorize implausible times (e.g., < 300 ms or > session length).
- **Cheating / forged scores — security hole:** `submit_attempt_v3` is `GRANT EXECUTE ... TO authenticated`, so via PostgREST a logged-in student can call the RPC **directly** with `p_actual_score := 1.0` and an arbitrary `p_reward`, bypassing `grade_answer()` entirely. The decision-consume check requires a valid unconsumed `decision_id` matching the question, but the **score itself is attacker-controlled**. A student can max their Elo/BKT on any recommended item. Server-side grading in the API is correct, but it's optional from the DB's perspective. Restrict the RPC to `service_role`, or grade inside the function from `answer_key` rather than trusting `p_actual_score`.
- **AI-help detection is silently broken:** `was_ai_used()` reads `decision.get("created_at")`, but `get_adaptive_decision()` does **not select `created_at`** → always `None` → function returns `False` every time. The K=0 "freeze on AI help" therefore never triggers from server detection; it only fires in stub mode from the (untrusted) client flag. Confirmed by inspection.
- **Idempotency replay path is broken:** In the early-return branch, `SELECT to_jsonb(qa) INTO attempt_id` assigns `jsonb` into a `uuid` variable — no such cast exists, so this raises instead of returning the cached result. The API-layer `consumed_at` check usually shields it, but a race between the API read and the RPC consume lands here and surfaces as a generic 500 (not the intended 409). Select a real column (`SELECT qa.id INTO attempt_id`).
- **Broken/missing prerequisite links:** `weight` is validated `[0,1]` at the API, but rows inserted by other paths or with `relation_type` typos are silently ignored (filtered out), so a mistyped relation just disables propagation with no warning. Self-loops (`source = target`) aren't rejected and would no-op via the `visited`/depth-1 logic but still burn a lock. Add a CHECK constraint `source_concept_id <> target_concept_id` and validate `relation_type` against an enum.
- **Transactions are theater in the Python layer:** `begin()/commit()/rollback()` all `return None`. The `db.begin()...db.commit()` wrapping in `/recommend` and `/submit` provides **no atomicity** across the multiple PostgREST calls. Only the single RPC is atomic. The pre-RPC reads (`get_student_mastery`, `get_question_by_id`, `count_hints`) are non-transactional w.r.t. the RPC, so `old_elo`/`old_bkt` shown to the user can be stale. Also `get_student_mastery` applies forgetting decay in Python **and** the RPC re-applies it from the raw stored value — decay is computed in two places with two code paths.

## 4. Actionable Refactoring Roadmap

Ordered by risk-reduction per unit effort. The guiding principle: **keep the synchronous transaction tiny and authoritative for the student; move calibration/learning off the hot path.**

### Tier 0 — Correctness & security hotfixes (days, do before any load)
1. **Lock down the RPC:** `REVOKE EXECUTE ... FROM authenticated; GRANT ... TO service_role;` and/or grade inside the function from `answer_key`. Closes the forged-score hole.
2. **Fix `was_ai_used`** (select `created_at`) and the **idempotency `INTO`** bug (`SELECT qa.id`).
3. **Clamp response time** (floor ~300 ms, cap to a plausible max) before it touches the speed factor.
4. **Cap `stability_days`** (e.g., 36500) and stop the unbounded doubling.
5. **Rename "FSRS" → "exponential forgetting (Woźniak/Ebbinghaus)"** in docs, or implement the real power curve. Fix the changelog β/γ mislabel and the Pelánek speed-weighting attribution.

### Tier 1 — Resolve the dual-implementation problem (the integrity fix)
6. **Choose one source of truth.** Either (a) make the SQL RPC the only implementation and **run the eval suite through it** (ephemeral Postgres in CI via testcontainers), or (b) keep Python authoritative and make the RPC a thin caller. Add an **equivalence test** asserting SQL and Python produce identical Elo/BKT/bandit outputs on a fixed vector set. Until this exists, no eval result describes production.
7. **Unify graph propagation:** pick depth-1 *or* recursive, in one place, and make it **order-invariant** (aggregate incoming deltas before applying).

### Tier 2 — Kill the hot-row bottleneck (the scalability fix)
8. **Make the student update the only synchronous write.** The student's mastery + the quiz_attempt insert + decision-consume are what the learner needs to see immediately and what needs strict idempotency. Keep *only* these in the synchronous transaction. Drop the `FOR UPDATE` on `questions` from the hot path.
9. **Asynchronous question calibration (batch Elo for items).** Question difficulty does not need per-submit, per-row-locked updates. Append `(question_id, score, expected, response_time)` to an event stream (outbox table / queue), and run a **batched calibrator** (every N seconds or N events) that updates `difficulty_elo` and `avg_response_time_ms` in bulk with a single short transaction. This removes the global per-question lock entirely and is *more* statistically stable (Pelánek explicitly supports periodic recalibration).
10. **Asynchronous bandit learning.** Move the Sherman–Morrison update out of the request. Append rewards to the outbox; a worker folds them into `A⁻¹`/`b` in `float8`, with **symmetrization + periodic re-inversion from `A`** and the `denom` guard fixed (skip both `A⁻¹` and `b`, or regularize). Arm *selection* can read a slightly stale `A⁻¹` from cache — bandits tolerate bounded staleness.
11. **Eliminate synchronous graph propagation.** Propagation is a derived, eventually-consistent view. Emit a "mastery changed" event and let a worker propagate depth-1/recursively with deterministic lock ordering (`ORDER BY concept_id`) outside the student's request.

After Tier 2, the synchronous submit touches **one student-owned row** + a couple of append-only inserts → no cross-student hot rows → it scales horizontally and the connection pool stops starving.

### Tier 3 — Performance & semantic caching
12. **Switch all DB matrix math to `double precision`.**
13. **Semantic / read caching:** you already have a write-through mastery cache; extend it to cache `recommend` candidate metadata and per-arm `A⁻¹` so selection avoids N per-arm round-trips (the current `/recommend` issues one `get_bandit_arm` per candidate — N queries per recommendation). Batch-load arms for a concept in one query.
14. **Backfill query in the migration** (`SET attempt_count = (SELECT count(*) ... )` correlated subquery) → rewrite as a single `UPDATE ... FROM (SELECT question_id, count(*) GROUP BY ...)` join.

### Tier 4 — Make the science publishable
15. Implement the §2.5 program: production-path eval, ≥30 seeds with CIs, next-step predictive AUC/Brier + reliability diagrams, true regret vs. oracle, misspecification tests, and at least one public-dataset validation.
